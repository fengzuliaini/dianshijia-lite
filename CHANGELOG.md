# 更新日志 (CHANGELOG)

本项目专注于为老电视及机顶盒（Android 4.1+, API 16+）与智能触屏手机提供完美的兼容性、流畅的遥控与触屏操作、以及极致的流媒体播放体验。

## [V1.4.0] - 2026-05-27
### 重构 (Refactored)
* **全面重构播放器引擎为内置 FFmpeg 解码的 IjkPlayer**：
  * **解决大部分源在老电视解析失败硬伤**：由于 ExoPlayer 深度依赖系统底层的 MediaCodec 硬件解码器，在 Android 4.4 等低配置老电视（如乐视 max70）上极易因为芯片不支持现代高清 Profile 或者是音频编码兼容缺陷导致初始化失败（报错“解析失败”）。我们彻底弃用了 ExoPlayer，重构并全量引入在智能电视上兼容性极佳的 **IjkPlayer (基于 FFmpeg 0.8.8 稳定版)**。
  * **FFmpeg 独立软解及硬解自动降级**：所有视频流协议的解复用（Demux）和音视频解码均在 C/C++ native 层由高度优化、强力容错的 FFmpeg 独立完成，不依赖电视系统底层的硬解芯片。同时配置硬解优先，在硬解初始化失败时自动无缝降级到软解，彻底救活大量高清 H.264/H.265 直播源。
  * **界面层全屏 TextureView 替换**：在 [activity_main.xml](file:///d:/开发/老电视直播软件/app/src/main/res/layout/activity_main.xml) 中将原有的 `PlayerView` 替换为原生的 `TextureView`，不仅提供了轻量级的渲染通路，还完美规避了 `SurfaceView` 对 OSD 菜单、侧边栏焦点的遮挡缺陷。
  * **IjkPlayer 电视端专属调优**：
    * 禁用 `packet-buffering` 缓冲区，省去多余的网络缓存延迟，实现换台瞬间秒开；
    * 限制 HLS 探测的 `analyzeduration` 为 1 秒，`probesize` 为 64KB，大幅提升首帧加载出画速度；
    * 禁用 `OpenSL ES`，改用系统兼容性最强的 `AudioTrack` 进行音频输出，适配老电视的陈旧音频驱动；
    * 挂接 `SurfaceTextureListener`，在 Surface 的 Available 和 Destroyed 阶段分别绑定和释放物理 Surface，保障生命周期与底层播放器的安全脱耦。
### 移除 (Removed)
* 彻底移除了 `com.google.android.exoplayer:exoplayer` (ExoPlayer 2.19.1) 全量依赖库，从而精炼了安装包结构，大幅缩减了老电视运存负载。

## [V1.3.9] - 2026-05-27
### 修复 (Fixed)
* **老旧电视（API <= 22）上由于过度代理引起普通 HTTP 源连接失败**：
  * **精准代理机制**：将本地代理 `proxyUrlIfNeeded` 拦截范围收窄为仅匹配 `https://` 加密协议的源，对普通的 `http://` 优质源（如中国移动 PLTV 高清流）放行直连。彻底消除因普通 HTTP 优质源被强行代理带来的 TCP 频繁握手、并发及代理层 I/O 瓶颈，恢复和 TVbox 相同的网络并发拉流与连接稳定性。
  * **回滚视频源**：恢复加载原有的 V1.3.7 网络 M3U 播放源（`http://us.3223516.xyz:55000/tv.m3u`），不再默认内置静态 `tv.txt` 直播源。
### 优化 (Improved)
* **播放器缓冲与解码容错优化**：
  * **低内存缓冲优化**：在初始化时配置 `DefaultLoadControl`，将最小缓冲区间从默认的 50 秒调低为更契合老旧电视低端硬件的 15 秒，极大地降低由于缓冲过大引发的 OOM（内存溢出）崩溃概率。
  * **解码器自动降级候选**：配置 `DefaultRenderersFactory` 开启扩展候选解码器支持（`EXTENSION_RENDERER_MODE_ON`），当老电视系统默认硬件解码器初始化或渲染出错时，允许播放器自动匹配降级到系统软件解码器（如 `OMX.google.h264.decoder` 等），保证视频流正常画面渲染。
* **升级开发工具与编译链至 SDK 34**：
  * **编译与目标 SDK 版本升级**：在保持 `minSdk 16` 允许老电视顺利安装的前提下，将 `compileSdk` 和 `targetSdk` 升级至最新主流稳定的 `34` (Android 14)，利用更高级的构建混淆工具生成在旧设备上执行更优化的字节码。

### 重构 (Refactored)
* **双核自适应直播源解析器**：
  * **智能分流解析**：重构 [M3uParser.java](file:///d:/开发/老电视直播软件/app/src/main/java/com/dianshijia/lite/parser/M3uParser.java)，增加对 TVbox/DIYP 风格的 TXT 直播源（以 `,#genre#` 作为分类标记，`#` 分割多源）的解析支持。解析器能自动检测输入文本特征，对于以 `#EXTM3U` 开头的源自动采用 M3U 协议解析，其余情况自动降级到 TXT 协议解析，实现平滑自适应。
  * **统一智能优先级排序**：将原有的“高清 PLTV 置顶、不支持的 H.265 源（针对老旧电视 API <= 22）沉底”的 `Collections.sort` 排序逻辑抽离为公共方法，对 M3U 和 TXT 解析出来的多线路源进行一致的智能重排，确保播放兼容度与画质的最佳平衡。

## [V1.3.6] - 2026-05-27
### 优化 (Improved)
* **寻回丢失直播线路并支持清晰度优先排序**：
  * **非排他性多源并入**：取消在解析 M3U 时对 `/TVOD/` 链接的排他性隔离。将所有切片流（包括 PLTV 和 TVOD）统一归入 `liveUrls`（直播线路列表），找回所有被隐藏和丢失的一半备用线路，大幅提升线路容灾基数。
  * **多维度智能排序算法**：引入 `Collections.sort` 对所有频道的直播线路进行优先级动态排序。普通设备上，清晰度高的 `/PLTV/` 纯直播源自动置顶优先播放，清晰度稍低的 `/TVOD/` 时移录制源排在后方充当容灾备用；老电视（API <= 22）上，则在此规则之上强制将所有不支持的 H.265 源（含 `h265/hevc` 关键字）沉到底部，实现清晰度、兼容度与线路完整度的多重智能优化。

## [V1.3.0] - 2026-05-26

### 🚀 全量 HTTP 本地代理——彻底修复老电视播放失败
* **HTTP/HTTPS 全量代理突破**：
  * 经数据分析发现所有 300 个直播源 URL 均为 `http://` 协议，此前四轮的 HTTPS-only 代理条件 (`url.startsWith("https://")`) 永远不满足，代理从未被触发。
  * 将 `proxyUrlIfNeeded()` 中的条件扩展为同时匹配 `http://` 和 `https://`，在老设备 (API ≤ 20) 上实现全协议全量代理。
* **302 重定向后相对路径 TS 分片支持**：
  * 分析发现上游服务器会将请求 302 重定向至带一次性 token 的 IP 地址，且 M3U8 内所有 TS 分片均为相对路径。
  * 重写 `rewriteM3u8Content()` 方法，利用 OkHttp `response.request().url()` 获取重定向后的最终 URL 作为 base，逐行解析 M3U8 内容，对绝对路径和相对路径统一改写为本地代理地址。
* **OkHttp 超时防护**：
  * 为代理 OkHttpClient 添加 `connectTimeout(15s)` 和 `readTimeout(30s)`，防止上游慢响应导致代理线程死锁。
